# Задание 1. REST API тестирование (Postman + Newman)

Коллекция покрывает CRUD-сценарии ресурса `/booking` публичного API **Restful Booker**
(`https://restful-booker.herokuapp.com`), а также авторизацию, негативные кейсы,
граничные значения дат и фильтрацию.

## Состав

| Файл | Назначение |
|------|------------|
| `collection.json` | Postman-коллекция со всеми запросами и тестами |
| `environment.json` | Environment с переменными (`base_url`, `token`, `booking_id` и др.) |

## Environment-переменные

| Переменная | Назначение |
|------------|-----------|
| `base_url` | Базовый URL |
| `token` | Токен авторизации, сохраняется автоматически после `POST /auth` |
| `booking_id` | ID бронирования из позитивного CRUD-сценария |
| `negative_booking_id` | ID бронирования для негативных проверок авторизации |
| `filter_*` | Уникальные значения для проверки фильтрации `GET /booking` |

Все переменные, кроме `base_url`, заполняются автоматически пре-реквест и тест-скриптами,
поэтому коллекция запускается «из коробки», без ручных правок.

## Запуск через Newman

Требуется Node.js 18+.

```bash
# установка Newman
npm install -g newman

# запуск коллекции с окружением
newman run collection.json -e environment.json
```

Без глобальной установки:

```bash
npx -y newman run collection.json -e environment.json
```

HTML-отчёт (опционально):

```bash
npm install -g newman-reporter-htmlextra
newman run collection.json -e environment.json -r htmlextra
```

## Покрытие

Коллекция разбита на 5 папок:

- **Auth** — получение токена (`POST /auth`) и проверка отказа при неверных кредах.
- **Booking positive CRUD** — полный жизненный цикл: создание → чтение списка →
  чтение по ID -> `PUT` -> `PATCH` -> `DELETE` → проверка удаления (404).
- **Booking negative tests** — несуществующий ID, отсутствие обязательных полей,
  неверные типы данных, запросы без токена и с невалидным токеном (403).
- **Boundary values - Dates** — корректные даты, прошедшая дата `checkin`,
  `checkout` раньше `checkin`, невалидный формат даты.
- **Filtering Booking** — фильтрация по `firstname` / `lastname` и по `checkin` / `checkout`.

Для каждого запроса есть assertions: статус-код, структура/схема JSON-ответа
(`pm.expect` / JSON-schema), время ответа (< 2000 ms).

Итог прогона: **23 запроса, 61 assertion — все проходят.**

## Найденные баги / наблюдения

1. **Слабая валидация типов данных.** `POST /booking` с неверными типами полей
   возвращает `500 Internal Server Error` вместо `400 Bad Request`.
2. **Нет валидации обязательных полей.** Неполный payload приводит к `500`,
   а не к `400`.
3. **Нет валидации дат.** API принимает прошедшую дату `checkin`, а также
   диапазон с `checkout` раньше `checkin` (например `2026-06-30 → 2026-06-01`)
   и возвращает `200 OK`.
4. **Неправильный формат даты сохраняется.** Невалидная дата сохраняется как
   `"0NaN-aN-aN"` вместо отклонения запроса.
5. **Нестандартный код при удалении.** `DELETE /booking/:id` возвращает
   `201 Created` вместо привычных `200 OK` / `204 No Content`
   (поведение задокументировано в API, проверка учитывает это).

## Что бы улучшил при наличии дополнительного времени

- Вынес бы повторяющиеся проверки (статус, время ответа) в коллекционные/папочные
  тест-скрипты, чтобы не дублировать их в каждом запросе.
- Добавил бы прогон в несколько итераций с data-файлом (`-d data.json`) для
  параметризованной проверки фильтрации и граничных дат.
- Подключил бы `newman-reporter-htmlextra` и автозапуск в CI (GitHub Actions).
